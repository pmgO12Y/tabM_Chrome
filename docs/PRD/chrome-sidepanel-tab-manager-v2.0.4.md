# Chrome 侧边栏标签管理插件 PRD v2.0.4

## Executive Summary
- 本版本在 `v2.0.3` 的自动定位能力基础上，优化侧边栏定位时的滚动体验。
- 目标是让切换 Chrome 顶部标签栏时，侧边栏使用平滑滑动定位，而不是瞬间跳到目标位置。

## Problem Statement
- `v2.0.3` 已经能正确定位到当前网页所属标签，但滚动行为仍然是即时跳转。
- 这种瞬间移动虽然功能正确，但视觉上会显得生硬，不利于用户跟上当前位置变化。
- 需要在不改变定位准确性的前提下，把后续标签切换改为平滑滚动。

## Target Users
- 希望侧边栏定位过程更自然、更容易跟上当前网页变化的用户。
- 已经依赖顶部标签点击后自动定位功能的用户。

## Solution Overview
- 保留首次进入侧边栏时的即时定位，避免一打开就缓慢滚动。
- 在后续切换顶部标签时，若当前目标标签发生变化，则改用 `scrollIntoView(..., behavior: "smooth")` 执行平滑滚动。
- 通过记录上一次定位的标签 key，区分“首次定位”和“后续切换定位”。

## Success Metrics
- 切换 Chrome 顶部标签时，侧边栏会平滑滑动到对应标签。
- 首次打开侧边栏时，不会出现不必要的长距离慢滚。
- 不影响现有自动定位准确性和高亮规则。

## User Stories / Acceptance Criteria
- 作为用户，我切换顶部标签时，侧边栏应平滑滑动到对应位置，而不是瞬间跳过去。
- 作为用户，我第一次打开侧边栏时，列表可以直接定位，不需要额外等待滚动动画。
- 作为用户，多窗口场景下，平滑滚动仍然只跟随当前窗口里的当前网页标签。

## Out of Scope
- 本版本不改变当前网页判定逻辑。
- 本版本不改变窗口排序。
- 本版本不改变高亮样式层级。

## Risks / Dependencies
- 平滑滚动仍依赖浏览器对 `scrollIntoView({ behavior: "smooth" })` 的支持。
- 如果后续列表渲染结构变化，需要同步确认记录的 `previousActiveRowKey` 仍然稳定。

## Open Questions
- 如果后续用户希望可以开关动画，可考虑单独引入“滚动风格”设置；本版本先固定为“首次即时，后续平滑”。
