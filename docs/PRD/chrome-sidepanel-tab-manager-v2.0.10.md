# Chrome 侧边栏标签管理插件 PRD v2.0.10

## Executive Summary
- 本版本进一步修正窗口行右侧信息区的实现方式，正式采用固定右侧槽位。
- 目标是让“当前”和数量不再依赖普通布局挤压或绝对定位补丁，而是拥有一块稳定、固定宽度的右侧区域。

## Problem Statement
- `v2.0.9` 虽然把右侧信息改成了绝对定位，但本质仍像是在现有布局上打补丁。
- 用户给出的宽窄对比截图表明，这种方式依旧没有准确满足“固定右侧槽位”的需求表达。
- 用户真正需要的是：窗口行天生分成左侧可省略标题区和右侧固定槽位区，而不是继续修补右侧信息的显示方式。

## Target Users
- 会频繁缩窄 Chrome 侧边栏的用户。
- 未来希望在窗口行右侧加入短按钮、状态标记或其他轻量操作的用户。

## Solution Overview
- 将窗口行恢复为结构明确的三段布局：箭头、标题、固定宽度右侧槽位。
- 右侧槽位宽度通过 CSS 变量统一控制，当前为“当前 + 数量”的稳定显示预留空间。
- 标题继续作为唯一允许被压缩和省略的区域。
- 右侧槽位内部继续采用单行右对齐，避免因为宽度变化而漂移。

## Success Metrics
- 宽窄两种侧边栏状态下，右侧“当前”和数量都能显示在固定槽位中。
- 标题在空间不足时被优先省略，右侧槽位不漂移、不消失。
- 后续若加入短按钮，可直接复用这块右侧槽位。

## User Stories / Acceptance Criteria
- 作为用户，我把侧边栏缩窄后，窗口右侧“当前”和数量仍应稳定存在。
- 作为用户，我希望右侧信息区像固定栏位一样稳定，而不是随着布局变化乱动。
- 作为用户，窗口标题可以省略，但右侧固定槽位应一直保留。

## Out of Scope
- 本版本不改标签列表的结构和交互。
- 本版本不改自动定位、排序和高亮逻辑。

## Risks / Dependencies
- 右侧固定槽位宽度需要覆盖当前内容和后续短按钮的展示需求。
- 若未来加入更长文案按钮，需要同步调整槽位宽度或限制文案长度。

## Open Questions
- 当前无新的产品决策问题；后续如新增右侧按钮，可继续沿用该固定槽位。
