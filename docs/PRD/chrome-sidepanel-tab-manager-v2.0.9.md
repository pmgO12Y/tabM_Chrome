# Chrome 侧边栏标签管理插件 PRD v2.0.9

## Executive Summary
- 本版本修正 `v2.0.8` 对窗口右侧信息区的实现方向错误。
- 目标是让窗口行右侧的“当前”和“数量”真正固定在最右侧，而不是继续作为普通布局列参与挤压。

## Problem Statement
- `v2.0.8` 虽然强调“右侧固定”，但实际实现仍然使用普通流式布局。
- 这意味着右侧信息区依然会参与宽度分配，在侧边栏缩到最窄时仍可能被裁掉。
- 用户需要的不是“尽量别被挤掉”，而是“像固定动作区一样钉死在最右侧”。

## Target Users
- 会把 Chrome 侧边栏缩到极窄宽度使用的用户。
- 希望后续在窗口行右侧继续加入短按钮或状态标记的用户。

## Solution Overview
- 把窗口行从 `grid` 三列布局改成普通单行容器。
- 将右侧 `.window-row__meta` 改为绝对定位，固定贴在窗口行最右侧。
- 窗口行本体通过右侧内边距预留信息区空间，中间标题只在剩余区域内省略。
- 右侧信息区继续保留“不换行、右对齐”的规则，并禁止拦截点击。

## Success Metrics
- 侧边栏缩到最窄时，“当前”和数量仍完整显示在最右侧。
- 标题继续在剩余空间里正常省略，不再把右侧信息挤没。
- 后续右侧加入短按钮时，可以继续复用同一块固定锚点区。

## User Stories / Acceptance Criteria
- 作为用户，我把侧边栏缩到极窄时，窗口右侧“当前”和数量仍然可见。
- 作为用户，我希望右侧信息像固定操作区一样始终停在最右边。
- 作为用户，窗口标题可以被省略，但右侧信息区不应该再消失。

## Out of Scope
- 本版本不改标签列表结构。
- 本版本不改自动定位、窗口排序、标签高亮和交互逻辑。

## Risks / Dependencies
- 右侧信息区预留宽度需要覆盖当前“当前 + 数量”的展示宽度。
- 若未来右侧加入更长文案按钮，需要同步扩大预留宽度或约束按钮长度。

## Open Questions
- 当前无新的产品决策问题；后续若增加右侧按钮，可继续沿用该固定锚点实现。
