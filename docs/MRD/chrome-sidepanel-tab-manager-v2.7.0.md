## 市场背景
| 维度 | 说明 |
| --- | --- |
| 使用环境 | Chrome 浏览器标签页数量持续增长，用户越来越依赖侧边栏、垂直标签和分窗口工作流进行信息管理。 |
| 用户趋势 | 重度浏览器用户不再满足于“看见标签”，而是需要“快速定位 + 快速整理 + 低打断切换”。 |
| 当前缺口 | 仅靠滚动浏览标签列表时，用户在几十到上百个标签中定位目标内容的效率明显下降。 |
| 机会点 | 在现有轻量高性能侧边栏基础上补齐搜索与结果处理能力，可以显著提升插件在高标签密度场景中的使用价值。 |

## 目标用户
| 用户类型 | 特征 | 主要诉求 |
| --- | --- | --- |
| 重度标签用户 | 同时打开大量标签，跨多个窗口切换 | 快速找到目标标签，避免反复滚动查找 |
| 任务并行用户 | 按主题或任务拆分浏览窗口 | 搜索后快速把相关标签转移到独立窗口 |
| 信息整理型用户 | 经常做标签分组、整理、归档 | 在不破坏既有工作流的前提下完成定位与再编排 |

## 典型场景
| 场景 | 用户行为 | 期望结果 |
| --- | --- | --- |
| 快速定位 | 用户记得页面标题关键词或 URL 片段 | 通过实时搜索迅速缩小范围 |
| 搜后整理 | 用户搜索出一批相关标签，希望单独处理 | 将命中标签一键移动到新窗口 |
| 浏览不中断 | 用户正在浏览长列表，不希望搜索入口随滚动消失 | 搜索框固定可见，随时可调用 |
| 键盘优先 | 用户习惯先键盘操作再鼠标确认 | `/` 快速聚焦，`Esc` 快速退出/清空 |

## 核心痛点
| 痛点 | 表现 | 影响 |
| --- | --- | --- |
| 定位成本高 | 标签多时只能人工滚动查找 | 查找时间长、认知负担高 |
| 找到后仍需重复整理 | 找到相关标签后还要逐个搬运或切换窗口 | 操作链条长、效率低 |
| 搜索入口不稳定会打断流程 | 若搜索入口随滚动离开视野，用户需要先回到某个位置 | 增加额外界面操作 |
| 复杂搜索不一定等于高价值 | 早期直接上高级筛选会增加理解与维护成本 | 偏离轻量高性能定位 |

## 需求机会
- 在不引入复杂搜索语法的前提下，优先满足最高频的“标题 / URL 快速定位”需求。
- 将“搜索结果”直接连接到“整理动作”，缩短从发现到处理的路径。
- 用过滤模式与高亮模式覆盖两类高频心智：
  - 只看结果
  - 保留全局上下文但弱化无关标签
- 通过固定搜索区与键盘快捷键，提升搜索功能的可达性与调用频率。

## 范围边界
### In Scope
- 标题 + URL 实时搜索
- 过滤模式与高亮模式切换
- 匹配结果数量反馈
- 无结果提示
- `/` 聚焦搜索、`Esc` 清空搜索
- 将命中标签移动到新窗口
- 搜索区图标与悬浮提示统一

### Out of Scope
- 正则搜索
- 多字段高级筛选
- 按窗口 / 分组 / 域名的结构化筛选器
- 搜索历史、最近搜索、保存搜索方案
- 搜索关键词在标题中的富高亮着色
- 用户自定义快捷键

## 成功指标 / 验收口径
| 维度 | 验收口径 |
| --- | --- |
| 搜索效率 | 用户输入关键词后，结果能即时反映到侧边栏展示状态 |
| 可理解性 | 用户能明确理解当前是过滤模式还是高亮模式 |
| 可操作性 | 搜索结果存在时，可直接执行“移动到新窗口”动作 |
| 可恢复性 | 用户可通过 `Esc` 或清除按钮迅速退出搜索状态 |
| 一致性 | 搜索能力不破坏现有标签、窗口、分组管理心智 |

## 风险假设
| 风险 / 假设 | 说明 | 应对策略 |
| --- | --- | --- |
| 用户需求继续扩张 | 上线后可能继续提出更复杂筛选诉求 | 当前版本先维持轻量方案，为后续版本预留扩展空间 |
| 搜索与列表渲染耦合 | 搜索态会影响列表的呈现方式 | 通过模式切换清晰区分“过滤”与“高亮” |
| 文档与发版节奏不一致 | 文档版本可能先于正式构建版本 | 在文档中明确当前属于规划 / 交付口径，不误写为已上线 |
